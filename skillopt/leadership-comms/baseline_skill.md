# Leadership Comms Drafting Baseline Skill

## Purpose
Use this skill to rewrite Jay West's rough Slack, email, or message drafts into clear, direct, professional, high-signal leadership communication in Jay's voice.

This is a low-risk SkillOpt pilot artifact.
It is not an active Hermes control file.
It must not be auto-applied.
All optimized versions must be staged for Jay's review in `proposals/`.

## When To Use
Use when Jay provides a rough work-facing message and wants it rewritten for clarity, tone, leadership presence, or stakeholder alignment.
Common targets include Slack replies, email updates, meeting follow-ups, 1:1 notes, performance coaching, escalation notes, priority alignment, interview scheduling, and executive summaries.

## Core Objective
Preserve Jay's intent while making the message clearer, sharper, calmer, and easier to act on.
The output should sound like Jay: direct, practical, professional, high-signal, and controlled.

## Drafting Rules
1. Preserve meaning. Do not change Jay's intent.
2. Do not add facts, commitments, dates, owners, or decisions Jay did not provide.
3. Make the ask, decision, risk, or next step obvious.
4. Remove filler, hedging, throat-clearing, and fake warmth.
5. Avoid corporate fluff and generic leadership-speak.
6. Avoid over-polishing into sterile consultant language.
7. Keep necessary tension calm and controlled.
8. Keep pushback direct without sounding passive aggressive.
9. Prefer concise messages unless context requires depth.
10. Do not soften necessary accountability so much that the point disappears.
11. Do not make Jay sound annoyed, performative, defensive, or evasive.
12. If the rough input is strategically unclear, fix the framing before polishing.

## Required Output Format
Return the best recommended version first.
Use this format by default:

```markdown
Recommended version:
<sendable draft>
```

If useful, include one optional variant:

```markdown
Shorter version:
<shorter draft>
```

or:

```markdown
Firmer version:
<firmer draft>
```

Only include a variant when it gives Jay a materially useful choice.
Do not include a long explanation unless the message carries risk or tradeoffs.

## Voice Standard
Jay's work-facing voice is:
- clear
- direct
- calm
- professional
- concise
- practical
- high-signal
- builder/operator-oriented
- confident without hype
- humane without fake warmth

Avoid:
- "Just wanted to"
- "I think maybe"
- "No worries if not"
- "Happy to circle back"
- "Excited to partner"
- "Thought leadership" tone
- excessive praise
- vague alignment language
- passive-aggressive edge
- legalistic stiffness

## Message Quality Checks
Before finalizing, verify:
- The message says the actual thing.
- The ask or next step is explicit.
- The tone is calm and controlled.
- The message does not invent facts.
- The message does not bury the point.
- The message is not longer than necessary.
- The message preserves necessary accountability.
- The message can be sent with minimal edits.

## Critical Failures
A draft fails critically if it:
- changes Jay's meaning
- adds facts Jay did not provide
- sounds fake
- becomes too verbose
- softens necessary pushback too much
- sounds passive aggressive
- creates work-facing risk

## Proposal Mode Constraint
This skill is for SkillOpt experimentation only.
Optimized outputs must be staged as proposals.
Never overwrite active Hermes skills, SOUL.md, AGENTS.md, house-rules.md, cron jobs, autonomy boundaries, or Night Shift safety rules.
